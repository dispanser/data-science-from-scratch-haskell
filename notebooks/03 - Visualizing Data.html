Visualizing Data¶

Line Charts: Nominal GDP¶

Let's define the data used in the line chart first. We add (optional) type signatures, and note that python lists are also valid haskell.

In [1]:
years :: [Int]
years = [1950, 1960, 1970, 1980, 1990, 2000, 2010]

gdp :: [Double]
gdp = [300.2, 543.3, 1075.9, 2862.5, 5979.6, 10289.7, 14958.3]

Chart: Graphics.Rendering.Chart.Easy¶

Let's try to recreate the line charts depicting the GDP by year.

In [2]:
import Graphics.Rendering.Chart.Easy

toRenderable $ do
  layout_title  .= "Nominal GDP"
  setColors [opaque blue, opaque green, opaque orange]
  layout_x_axis . laxis_title .= "year"
  layout_y_axis . laxis_title .= "Billions of $"
  plot $ line "gdp" [zip years (map (*0.8) gdp), zip years (map (*1.2) gdp)] -- two blue lines in chart
  plot $ line "gdp red" [zip years gdp] -- the green line (our real data)
  plot $ points "gdp points" $ zip years gdp

In contrast to matplotlib, the line and the points (marker = 'o' in the book) are plotted separately. Points must be plotted after the line to be plotted on top. Additionally, note that the line method accepts a list of lists of tuples, so we can draw multiple lines using the same color and legend entry. Might become useful for e.g. confidence intervals etc.

The monadic approach used for charting looks similar to the python code in the book. The increased line count can be attributed to:

  • explicitely drawing the points as a separate plot
  • setting the colors in a separate call (independent of the actual line chart)
  • adding an x-axis legend and
  • adding some additional lines for fun and profit.

On the plus side, Chart produces a legend for free.

Non-monadic Style¶

We can use the Graphics.Rendering.Chart module directly, providing more control. See charts documentation example 1. The pie chart example is not part of the book, and is certainly not a good way to represent this kind of data, but there are no pie charts in the visualization chapter, so let's just make one up!

In [3]:
import Graphics.Rendering.Chart
import Data.Default.Class
import Control.Lens

pitem year gdp = pitem_value .~ gdp
  $ pitem_label .~ show year
  $ pitem_offset .~ 3
  $ def

-- the full-blown chart api
toRenderable 
  $ pie_title .~ "GDP by year"
  $ pie_plot . pie_data .~ zipWith pitem years gdp
  $ def

Bar Charts: Hollywood¶

It was surprisingly hard to figure out how to properly format and align the x-axis to show string labels in a similar way to the Hollywood movie example in the book. Let's get started!

In [4]:
-- define the data
movies = ["Annie Hall", "Ben-Hur", "Casablanca", "Gandhi", "West Side Story"]
num_oscars = [5, 11, 3, 8, 10]

-- a bar plot is a list of tuples, the first tuple element being the position of the
-- bar (x-axis), snd is a list of values, representing the bar heights.
barv idx val = (idx, [val]) -- each entry in num_oscars is a singleton list inside a tuple
barvs = zipWith barv [0..] num_oscars
In [5]:
-- our first bar chart
toRenderable $ plot $ plotBars <$> bars movies barvs

:t bars
bars :: forall y x l. (BarsPlotValue y, PlotValue x) => [String] -> [(x, [y])] -> EC l (PlotBars x y)

Success! There are bars, right? They even have the proper height. Still, some things are off:

  • the first and last bar are not completely inside the view port
  • axis labels are completely useless and not related to the actual data we want to show
  • the legend shows the movie names, but the colors are not related to the bars on the chart

What's wrong? The [String] passed in as first argument to bars does not define the names for the individual bars. Instead, it is meant to be used when multiple bars are located at the same position. Additionally, we define values in the range [0 .. 4], and this becomes the range that is covered by the default x-axis.

In [6]:
barValues :: [(String, [Int])]
barValues = zip movies $ map (:[]) num_oscars

toRenderable $ do
  layout_title .= "My Favorite Movies"
  layout_y_axis . laxis_title .= "# of Acadamy Awards"
  layout_x_axis . laxis_generate .= autoIndexAxis (map fst barValues)
  layout_x_axis . laxis_style .= (axis_label_style . font_size .~ 8 $ def)
  plot $ plotBars <$> bars [ "" ] (addIndexes $ map snd barValues)

That looks a lot better. The bars are completely inside the viewport, and each bar is labelled with the name of the movie:

  • we use autoIndexAxis with the movie names to replace the x-axis representation with a labelled one
  • the font size is reduced to 8 to have enough room for each movie name. If the font is too large (or the text too long) not every label is visible.
  • we pass in a singleton list with an empty string as the bar legend to suppress the legend. An empty list does not work, the result is an empty chart without bars.
In [7]:
toRenderable $ do
  layout_title .= "Bollywool Oscars"
  layout_x_axis . laxis_override .= axisLabelsOverride (zip [0..] movies) 
  plot $ plotBars <$> bars ["oscars" ] barvs

Another attempt: replaced lambda and autoIndexAxis with axisLabelsOverride which seems to be exactly what we want: modify the existing AxisData by replacing the default labes with the ones passed in. However, the resulting x-axis again doesn't cover the range to fully contain all bars. Thinking about it, we took an existing axis definition and replaced the labels, so it's not surprising that this doesn't work.

Scatterplots: mind your axes¶

This example illustrates the effects of having comparable types on both the x-axis and the y-axes, without also providing similar value ranges for them.

In [16]:
test1Grades, test2Grades :: Num a => [a]
test1Grades = [ 99, 90, 85, 97, 80]
test2Grades = [100, 85, 60, 90, 70]

scatterplotUnequalAxes = toRenderable $ do
  layout_title .= "Axes Aren't Comparable"
  plot $ points "XYZ" $ zip (test1Grades :: [Int]) (test2Grades :: [Int])

scatterplotUnequalAxes

Well... almost. Some of the points are directly on either the grid or the ticks, making them almost hidden. This seems to be a general theme: Chart produces axis whose range seems to be rounded to the nearest round number (here: x-axis [80, 100], y-axis [60, 100]). However, it does not incorporate some minimal gap between the axis range and the value range, so values can lead exactly on the axis.

We try to improve on this chart by adding a hidden plot underneath it, with slightly increased value ranges to force all points to be strictly inside the viewport.

In [17]:
data AxisRange a = AxisRange a a deriving (Show, Eq)

-- scatter plot combining a point plot with a hidden plot to enlarge the
-- viewport somewhat, otherwise some points lay directly on the axis and ticks.
scatterplotUnequalAxes :: Renderable ()
scatterplotUnequalAxes = toRenderable unequalScatterLayout
  where
    AxisRange minX maxX = scaleRange 1.1 $ computeRange (test1Grades :: [Double])
    AxisRange minY maxY = scaleRange 1.1 $ computeRange (test2Grades :: [Double])
    hidden = PlotHidden { _plot_hidden_x_values = [minX, maxX]
                        , _plot_hidden_y_values = [minY, maxY] }
    pointPlot = plot_points_values .~ zip (test1Grades :: [Double]) (test2Grades :: [Double]) $
                plot_points_style . point_radius .~ 2 $
                def
    unequalScatterLayout = layout_title .~ "Axes Aren't Comparable" $
                           layout_x_axis . laxis_title .~ "test 1 grade" $
                           layout_y_axis . laxis_title .~ "test 2 grade" $
                           layout_plots .~ [toPlot pointPlot, toPlot hidden] $ def

-- | produce a range that scaled by `factor`.
scaleRange :: Fractional a => a -> AxisRange a -> AxisRange a
scaleRange factor (AxisRange minV maxV) =
  let middle = (maxV + minV) / 2
  in AxisRange (middle - (middle - minV)*factor) (middle + (maxV - middle)*factor)

computeRange :: Ord a => [a] -> AxisRange a
computeRange xs = AxisRange (minimum xs) (maximum xs)

scatterplotUnequalAxes

Looks okay-ish. There's substantial whitespace on both axis, but that's a feature of the underlying axis generator logic. As a next step, we make the axis ranges equal, which is unfortunately a lot more involved than matplotlib's plt.axis("equal").

In [18]:
-- | given two ranges, produce new set of ranges that have equal width
alignRanges :: (Fractional a, Ord a) => AxisRange a -> AxisRange a -> (AxisRange a, AxisRange a)
alignRanges x@(AxisRange minX maxX) y@(AxisRange minY maxY)=
  let
    xRange = maxX - minX
    yRange = maxY - minY
    range  = max xRange yRange
    newXrange = if range == xRange then x else scaleRange (range / xRange) x
    newYrange = if range == yRange then y else scaleRange (range / yRange) y
  in (newXrange, newYrange)

scatterplotEqualAxes :: Renderable ()
scatterplotEqualAxes = toRenderable layout
  where
    xRange = scaleRange 1.1 $ computeRange (test1Grades :: [Double])
    yRange = scaleRange 1.1 $ computeRange (test2Grades :: [Double])
    (AxisRange minX maxX, AxisRange minY maxY) = alignRanges xRange yRange
    hidden = PlotHidden { _plot_hidden_x_values = [minX, maxX]
                        , _plot_hidden_y_values = [minY, maxY] }
    pointPlot = plot_points_values .~ zip (test1Grades :: [Double]) (test2Grades :: [Double]) $
                plot_points_style . point_radius .~ 2 $
                def
    layout = layout_title .~ "Axes Are Almost Comparable" $
             layout_x_axis . laxis_title .~ "test 1 grade" $
             layout_y_axis . laxis_title .~ "test 2 grade" $
             layout_plots .~ [toPlot pointPlot, toPlot hidden] $ def

scatterplotEqualAxes

Quite a bit of code for such a basic requirement. The x-axis and the y-axis cover the same range, but they don't have the same scaling, because the plot itself is wider than tall. Using PlotHidden in combination with scaleRange could be used to manually adjust the axis to your liking.
